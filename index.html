多 Agent 团队配置

这个仓库不是单个 Agent prompt 的集合，而是一套完整的研发组织协议。它将产品、设计、架构、审批、实现、评审、安全、测试、发布拆成明确角色，并用统一 artifact 和 gate 规则保证每一步都可追溯、可审查、可交接。

项目定位 为企业软件与 AI 功能交付提供一套可执行的多 Agent 研发运行规范。
  • 统一产物：使用固定 artifact 标签管理交付
  • 明确门禁：未满足输入不得越级进入实现
  • 职责边界：关键决策只能由唯一角色负责
  • 可追溯性：每段实现都可回溯到需求与设计依据
当前仓库包含 项目基本结构。
  • AGENTS.md 多 Agent 协议总规约
  • .codex/agents/*.toml 各角色配置
  • docs/ 角色目录与模板文档
  • README.md 项目说明与使用方式
角色矩阵 围绕标准研发链路构建角色，而不是围绕单次提示词堆叠能力。

Product

由 senior-ai-agent-pm 负责 PRD、User Story、Prototype，确保目标、范围、验收标准先于实现存在。

PRD UserStory Prototype

Design

由 senior-ui-designer 负责 DesignSpec、UICode、InteractionSpec，避免进入实现后再反向补设计。

DesignSpec UICode InteractionSpec

Architecture

由 tech-lead-architect 输出 SystemArch、DBDesign、APIDoc、TaskBreakdown，把实现所需契约一次性定义清楚。

SystemArch DBDesign APIDoc

Approval

由 engineering-manager 承担唯一编码门禁，审批不通过时，前后端与 AI 实现角色全部必须阻塞。

Approval Gate

Implementation

前端、后端、AI 工程角色分别执行，实现前必须先产出 ImplementationPlan，而不是直接改代码。

FrontendCode BackendCode AICode

Quality

code-reviewer、security-engineer、senior-qa-engineer 和 devops-engineer 共同完成上线前闭环。

Review Security QA Release
协作流程 每个阶段只有在入口条件满足时才能进入下一步，否则按 BLOCKED 模板回退给上游。
01 Product

需求澄清后输出 PRD、UserStory、Prototype。

02 Design & Arch

设计与架构角色补齐 UI 规范、接口契约、任务拆解与系统边界。

03 Approval

工程经理基于完整输入决定 APPROVED 或 REJECTED。

04 Implement

实现角色先写计划，再交付前端、后端或 AI 代码产物。

05 Review

代码评审和系统评审识别 blocker、high、medium、low 风险。

06 Security

安全角色审查权限、依赖、输入校验与暴露面。

07 QA

测试角色输出用例与测试报告，确认关键路径和异常流。

08 Release

DevOps 在质量门禁通过后给出部署、回滚和监控方案。
