## 1. 背景与问题

Excel 需求要求应用管理承担“统一应用台账 + 项目关联 + 支撑资源关联 + 运行态感知 + 异常通知 + 责任边界管理”的核心职责，覆盖皖通公司本部及各营运单位的统一门户、业务平台、业务系统等信息化系统。

当前系统已有基础应用台账 CRUD 能力，但仍存在以下差距：

- 现状仅支持基础字段维护，尚未形成“应用为中心”的全景信息视图
- 现状只有 `projectId` 与若干责任人字段，缺乏服务器、数据库、API、网络资源等结构化依赖关系维护能力
- 现状支持简单状态字段，但未形成与 IT 资源、网络状态联动的应用运行态展示
- 现状没有应用异常通知、待办处理与影响范围追踪能力
- 现状没有围绕“业主方负责人、应用运维、管理团队负责人”定义应用域权限边界
- 现状统计分析不足，无法支撑“自建、集团统建、交通部统建”等分级分类管理视图

因此需要将应用管理从“基础台账录入页”升级为“应用全生命周期治理入口”。

## 2. 产品目标

- 建立统一、完整、可追溯的应用台账，沉淀应用基本信息、责任边界和归属关系
- 建立应用与项目、IT 资源、数据库、API、网络资源之间的依赖关系网络
- 支撑按建设来源、分类分级、运行状态、责任组织等维度开展应用治理和统计分析
- 将底层 IT 资源与网络异常转译为应用视角的运行告警和责任通知
- 明确应用域角色权限，保障不同角色“看什么、改什么、批什么”边界清晰

## 3. 范围

### 3.1 In Scope

- 应用台账登记与维护
- 应用基础信息、归属单位、责任团队、责任人维护
- 应用与项目的关联维护
- 应用与 IT 资源、数据库、API、网络资源的依赖关系维护
- 应用分级分类管理与统计分析
- 应用运行状态汇总展示
- 应用异常通知与待办联动
- 应用域角色权限管理

### 3.2 Out of Scope

- 统一认证中心本身的能力建设与用户目录治理实现
- 监控平台、短信平台、CMDB、API 网关等外部系统的技术实现
- 应用性能探针、链路追踪、日志采集等底层监控技术方案
- 跨系统主数据同步机制的技术实现

## 4. 关键角色

- 应用系统管理员：维护应用台账、依赖关系、分类分级与责任边界
- 应用运维人员：处理应用异常、确认底层资源影响、维护运行状态备注
- 业主方负责人：确认应用归属、业务责任、重要通知接收与升级处理
- 开发单位负责人：维护厂商侧责任信息、协同处理版本与依赖变更
- 统一认证团队负责人：负责应用接入统一认证相关责任边界
- IT 资源团队负责人：负责应用支撑资源协同与异常联动
- 数据资源团队负责人：负责应用所依赖数据资源接口与数据责任边界
- 网络安全团队负责人：负责网络与安全侧协同、异常定位与通知处理
- 管理层：查看应用分布、运行风险与分类统计分析

## 5. 现状与目标能力差距

### 5.1 台账维度

- 现状：仅有基础字段表单
- 目标：形成“基础信息 + 建设来源 + 归属组织 + 责任矩阵 + 外部访问 + 技术栈 + 依赖摘要”的统一台账

### 5.2 关系维度

- 现状：只有 `projectId` 和文本型 `relationSummary`
- 目标：可维护结构化依赖对象，至少区分项目、服务器/虚拟机、数据库/API、网络资源四类

### 5.3 运行维度

- 现状：应用状态手工维护
- 目标：展示应用自身运行状态、底层 IT 资源状态、网络状态及最近异常

### 5.4 协同维度

- 现状：无应用异常通知与跨团队协同入口
- 目标：异常能够通知到对应责任人，并沉淀待办与处理留痕

### 5.5 治理维度

- 现状：缺少分级分类视角和应用域权限边界
- 目标：支持按建设来源、应用类型、运行状态、归属单位、责任团队进行治理分析

## 6. 关键业务流程

### 6.1 应用台账登记

1. 应用系统管理员新建应用
2. 填写基础属性、建设来源、归属单位、访问地址、技术栈
3. 维护业主方负责人、开发单位负责人、统一认证团队负责人、IT 资源负责人、数据负责人、网络安全负责人
4. 选择所属项目或建立项目关联
5. 保存后生成应用台账记录

### 6.2 应用依赖关系建立

1. 在应用详情中进入“依赖关系”页签
2. 关联支撑服务器/虚拟机、数据库/API、网络资源、所属项目
3. 系统生成依赖清单与影响范围摘要
4. 后续依赖对象状态变化时，应用侧可同步感知

### 6.3 应用运行监测与异常联动

1. 系统汇总应用关联 IT 资源与网络资源状态
2. 若底层对象出现异常，则映射为应用告警
3. 生成异常消息，推送短信或待办给责任人
4. 应用运维人员进入应用详情查看影响对象和处理记录

### 6.4 应用治理分析

1. 管理层或应用管理员进入统计分析页
2. 按建设来源、应用分类、分级、归属单位、运行状态统计
3. 查看异常应用数量、风险分布、责任团队分布等结果

## 7. 信息对象定义

### 7.1 应用基础信息

- 应用编码
- 应用名称
- 应用类型
- 建设来源/分类分级
- 运行状态
- 所属项目
- 所属单位
- 访问地址
- 技术栈

### 7.2 责任矩阵

- 业主方负责人
- 开发单位名称
- 开发单位负责人
- 统一认证团队负责人
- IT 资源团队负责人
- 数据资源团队负责人
- 网络安全团队负责人

### 7.3 依赖关系

- 所属项目
- 支撑服务器/虚拟机
- 数据库或 API
- 网络资源
- 依赖说明

### 7.4 运行视图

- 应用当前状态
- 底层 IT 资源状态汇总
- 网络状态汇总
- 最近异常事件
- 最近通知记录

## 8. 状态模型

建议应用主状态至少包括：

- `BUILDING`：建设中
- `RUNNING`：运行中
- `ALERT`：告警中
- `DISABLED`：停用

补充规则：

- 当底层资源或网络存在未恢复异常时，可将应用运行视图标记为“告警中”
- `DISABLED` 应用默认不再参与常规运行告警统计，但应保留历史记录查询
- 应用主状态允许人工维护，联动状态作为展示与预警依据，不强制完全替代人工判断

## 9. 业务规则

- 每个应用必须具备唯一应用编码
- 每个应用必须至少维护业主方负责人或应用运维责任人
- 每个应用必须可关联 0..1 个主项目，后续可扩展多项目关系
- 每个应用必须支持维护至少一类依赖关系，若暂无结构化依赖，也必须填写依赖摘要
- 建设来源至少支持：自建、集团统建、交通部统建
- 应用异常消息必须能定位到责任团队或责任人
- 角色权限必须覆盖查看、编辑、通知处理、统计查看四类基础控制
- 删除应用时需要校验是否仍被项目、资源、监控或通知记录引用

## 10. 非功能与体验要求

- 列表页支持按名称、建设来源、分类分级、状态、所属单位、责任人筛选
- 详情页必须支持页签式查看基础信息、责任矩阵、依赖关系、运行概览、通知记录
- 应用依赖与异常信息需尽量以可读、可追踪方式呈现，避免仅使用长文本
- 告警信息必须包含影响对象、发生时间、责任人、处理状态
- 统计页面需支持按时间、单位、建设来源、状态维度切换

## 11. 验收标准

- 能为每个应用登记基础信息、归属单位、责任矩阵和建设来源
- 能维护应用与项目、IT 资源、数据库/API、网络资源之间的关联关系
- 能按自建、集团统建、交通部统建等维度查询和统计应用台账
- 能在应用详情中看到底层 IT 资源状态与网络状态摘要
- 当底层资源或网络异常时，能向相应责任人生成通知或待办
- 能基于应用域角色限制不同用户的查看与操作范围

## 12. 待架构澄清项

- 应用依赖关系是新建专门关系表，还是复用现有资源/项目关系模型
- 数据库/API 是否纳入统一“依赖对象”模型，还是先以摘要方式存储
- 应用告警的状态来源是拉取监控平台、被动接收事件，还是手工同步
- 短信通知、站内待办、企业通讯工具三类通知渠道的优先级和可用性
- “统一认证团队负责人”是否需要单独建模为专属责任角色字段或责任组
