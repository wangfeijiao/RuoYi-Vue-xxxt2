### 一、信息架构

建议 IT 资源管理模块采用两层结构：

1. 顶部看板区
展示资源总量、使用中、空闲、待审批、待执行、已回收。

2. 主体区域
Tab 1：资源台账
Tab 2：资源工单

### 二、页面一：资源台账列表页

#### 1. 页面目标

让管理员快速查看资源现状、筛选资源、进入详情、执行新增或变更入口。

#### 2. 筛选区字段

- 关键字
- 资源类型
- 资源状态
- 监控状态
- 所属项目
- 所属部门
- 责任人
- 交付时间范围

#### 3. 列表字段

- 资源编码
- 资源名称
- 资源类型
- 资源状态
- 监控状态
- 所属项目
- 责任人
- 维护人
- CPU/内存/存储摘要
- IP 地址
- 交付时间
- 更新时间
- 操作列

#### 4. 操作列

- 详情
- 编辑
- 发起变更
- 发起回收
- 删除/停用

#### 5. 交互要求

1. 状态字段用标签显示。
2. 监控异常使用醒目颜色。
3. 列表支持分页。
4. 支持导出。
5. 点击资源名称进入详情抽屉。

#### 6. 空态

文案建议：
- 暂无资源数据，请先新增资源或完成资源交付。

#### 7. 错态

文案建议：
- 资源列表加载失败，请刷新后重试。
- 当前筛选条件下无匹配资源。

### 三、页面二：资源详情抽屉

#### 1. 展示分区

1. 基础信息
资源编码、名称、类型、状态、监控状态。

2. 归属信息
项目、部门、责任人、维护人。

3. 配置信息
CPU、内存、存储、OS、IP、软件名称、软件版本、许可证数量。

4. 摘要信息
关联说明、性能摘要、备注。

5. 历史信息
创建时间、更新时间、最近一次相关工单。

#### 2. 交互要求

1. 详情抽屉只读展示。
2. 详情中提供发起变更、发起回收快捷入口。
3. 详情中可跳转查看关联工单。

### 四、页面三：新增/编辑资源弹窗

#### 1. 表单字段

- 资源编码
- 资源名称
- 资源类型
- 所属项目
- 所属部门
- 责任人
- 维护人
- CPU 核数
- 内存
- 存储
- 操作系统
- IP 地址
- 软件名称
- 软件版本
- 许可证数量
- 交付时间
- 关联说明
- 性能摘要
- 备注

#### 2. 表单规则

1. 资源编码必填且唯一。
2. 资源名称必填。
3. 资源类型必填。
4. 若填写 IP，则校验格式。
5. CPU、内存、存储必须为非负整数。
6. 编辑模式下，核心字段修改提示建议通过变更工单处理。

#### 3. 提交反馈

1. 成功：提示保存成功并刷新列表。
2. 失败：保留表单内容并高亮错误项。

### 六、页面五：发起资源申请弹窗

#### 1. 使用入口

- 工单页新建申请
- 项目上下文中的申请资源

#### 2. 表单字段

- 项目
- 申请标题
- 资源类型
- 期望 CPU
- 期望内存
- 期望存储
- 期望操作系统
- 期望软件/版本
- 期望完成时间

### 七、页面六：发起变更/回收弹窗

#### 1. 发起变更

字段建议：
- 资源对象
- 变更项说明
- 目标配置
- 期望完成时间
- 变更原因

交互要求：
1. 自动带出当前资源快照。
2. 展示变更前/申请后对照区。

#### 2. 发起回收

字段建议：
- 资源对象
- 回收原因
- 是否保留项目绑定历史
- 期望回收时间

交互要求：
1. 自动显示当前归属项目和责任人。
2. 已回收资源不可重复回收。

### 八、页面七：审批弹窗

#### 1. 展示内容

- 工单基础信息
- 项目信息
- 资源快照或申请载荷
- 申请原因
- 期望完成时间

#### 2. 操作区

- 审批通过
- 审批拒绝
- 审批意见

#### 3. 规则

1. 拒绝时意见必填。
2. 通过后进入待执行。
3. 审批完成后不可重复审批。

### 九、页面八：执行弹窗

#### 1. 执行申请工单

字段建议：
- 实际资源编码
- 实际资源名称
- 实际配置
- 实际 IP
- 交付时间
- 执行结果

#### 2. 执行变更工单

字段建议：
- 实际变更结果
- 执行说明
- 完成时间

#### 3. 执行回收工单

字段建议：
- 回收结果
- 数据清理说明
- 资源去向
- 完成时间

#### 4. 规则

1. 执行结果必填。
2. 完成后同步更新台账状态。
3. 执行后工单不可再次执行。

### 十、状态反馈与视觉规则

1. `PENDING` 使用待处理色。
2. `PENDING_EXECUTION` 使用提醒色。
3. `COMPLETED` 使用成功色。
4. `REJECTED` 使用失败色。
5. `CRITICAL` 监控状态需强提示。
6. 已回收资源在列表中弱化显示。

### 十一、异常流原型说明

1. 表单校验失败
表现：字段级错误提示，不关闭弹窗。

2. 状态不合法
表现：按钮禁用或点击后提示当前状态不支持该操作。

3. 后端保存失败
表现：顶部消息提醒 + 保留当前输入。

4. 数据不存在
表现：提示数据已被删除或状态已变化，请刷新后重试。

5. 权限不足
表现：隐藏无权限按钮；接口兜底返回权限错误提示。

### 十二、权限原型约束

1. 项目经理
可发起申请、查看本人或本项目工单、查看项目下资源。

2. IT 资源管理员
可查看全部资源、编辑台账、审批工单、发起变更和回收。

3. 运维执行人
可查看待执行工单、执行交付、变更、回收。

4. 管理层
只读查看统计和台账。

### 十三、建议的后续页面优化方向

1. 资源详情从抽屉升级为详情页，承载更多历史记录。
2. 工单详情增加时间轴。
3. 增加变更前后差异对比卡片。
4. 增加资源与项目的关系视图。
5. 增加超时提醒和待办角标。

### 建议新增/优化点

1. 收口资源状态和工单状态字典，避免前后端自由文本导致状态失控。
2. 将资源表单改成通用字段加类型专属字段机制，减少服务器、存储、许可证共用一套字段的混乱。
3. 增加空闲资源池概念，允许申请时优先绑定库存而不是总是新建资源。
4. 给工单补充撤回或作废能力，否则被误提交流程会长期滞留。
5. 增加变更差异视图，审批人和执行人都应直接看到变更前后对比。
6. 对回收流程增加检查项，如数据备份、账号清理、许可证释放，避免只改状态不做真实回收。
7. 增加资源到期、许可证到期、长期空闲资源的预警规则。
8. 增加批量导入和批量回收能力，否则台账维护成本会过高。
9. 增加资源与应用、网络、数据资产的关联摘要，为后续跨域分析做准备。
10. 将监控状态从摘要字段逐步升级为外部监控集成，避免长期依赖人工维护。
