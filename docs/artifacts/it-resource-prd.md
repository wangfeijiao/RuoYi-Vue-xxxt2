### 一、产品目标

1. 建立统一的 IT 资源台账，沉淀服务器、虚拟机、存储、软件、中间件、许可证等资源的完整生命周期记录。
2. 建立资源申请、审批、交付、变更、回收的闭环工单体系，替代线下沟通和口头确认。
3. 将资源与项目、责任部门、维护人建立明确关系，支撑项目建设、运行和验收。
4. 让资源状态、监控状态、交付进度、审批进度可查询、可追踪、可统计。
5. 为后续资源池、容量管理、成本核算、自动化交付预留产品结构。

### 二、业务边界

#### 1. In Scope

1. IT 资源台账登记、编辑、查询、导出。
2. 资源申请工单。
3. 资源变更工单。
4. 资源回收工单。
5. 审批与执行闭环。
6. 资源与项目关联。
7. 资源状态与监控状态展示。
8. 资源统计看板。
9. 权限控制与操作留痕。

#### 2. Out of Scope

1. 真实云平台、虚拟化平台、CMDB、监控平台的自动同步实现。
2. 采购流程、资产折旧、财务核算。
3. 网络资源、数据资源、应用关系图的完整交叉域管理。
4. 自动化部署脚本执行、主机探测、自动巡检。
5. 多级复杂审批编排引擎，MVP 仅采用标准单审批加执行流。

### 三、目标用户与角色

1. 项目经理
职责：发起资源申请、跟踪审批和交付、查看项目下资源清单。

2. IT 资源管理员
职责：维护资源台账、审批资源工单、确认资源分配策略、统筹回收。

3. 运维执行人
职责：执行资源交付、部署、扩容、配置调整、回收，并回填执行结果。

4. 部门负责人
职责：对关键资源申请、变更、回收进行业务确认。
说明：MVP 中可先弱化为扩展角色，不强制进入流程。

5. 管理层
职责：查看资源总量、使用状态、回收情况、异常情况、项目资源分布。

6. 审计或管理员
职责：查看操作记录、审批记录、执行留痕。

### 五、业务规则

1. 每条资源台账必须有唯一资源编码。
2. 同一有效状态下，IP 地址不允许重复绑定到多个运行中的资源。
3. 资源申请、变更、回收必须通过工单留痕，不允许直接改核心状态。
4. 资源台账的核心字段变更必须通过变更工单执行落账。
5. 资源的监控状态与业务状态分离。
6. 删除不应物理删除已被工单引用的资源，建议采用逻辑停用或已回收状态。
7. 回收后的资源默认不可被项目继续引用。
8. 审批拒绝必须记录理由。
9. 执行完成必须记录执行结果。
10. 每笔工单都必须能追溯申请人、审批人、执行人和时间戳。

### 六、主流程

#### 1. 资源台账登记流程

1. IT 资源管理员进入资源台账页面。
2. 录入资源基础信息、配置、责任人、所属项目。
3. 系统校验资源编码、IP 等关键字段。
4. 保存成功后形成可查询台账。
5. 默认状态为 `IDLE` 或按业务规则初始化。

#### 2. 资源申请流程

1. 项目经理选择项目并发起资源申请。
2. 填写资源类型、用途、期望配置、期望完成时间、申请原因。
3. 系统生成资源申请工单，状态为 `PENDING`。
4. IT 资源管理员审批。
5. 审批通过后工单进入 `PENDING_EXECUTION`。
6. 运维执行人完成交付并回填实际配置和交付结果。
7. 资源台账创建或绑定已有资源，资源状态进入 `IN_USE`。
8. 工单状态更新为 `COMPLETED`。

#### 3. 资源变更流程

1. 使用方或管理员对已有资源发起变更工单。
2. 填写变更项，如 CPU、内存、存储、软件版本、责任人。
3. 系统保存当前快照和申请载荷。
4. 审批通过后由运维执行。
5. 执行完成后更新台账核心字段。
6. 工单完成，保留变更前后痕迹。

#### 4. 资源回收流程

1. 项目结束或资源闲置时发起回收工单。
2. 审批确认后进入执行。
3. 运维执行回收、清理配置、回收账号或许可证。
4. 台账状态更新为 `RECYCLED` 或回归 `IDLE`。
5. 与项目解绑或标记历史归属。

### 七、异常流

1. 资源编码重复
处理：禁止保存，提示已有资源编码。

2. IP 地址冲突
处理：禁止提交或要求改为无 IP 资源类型。

3. 审批拒绝
处理：工单进入 `REJECTED`，必须填写拒绝意见，申请人可重新发起。

4. 执行失败
处理：工单保留在 `PENDING_EXECUTION` 或新增失败结果说明；MVP 可通过执行结果文本说明并允许再次执行。

5. 找不到目标资源
场景：变更或回收时引用资源不存在或已停用。
处理：禁止提交，提示刷新台账。

6. 资源已处于不可操作状态
场景：对已回收资源发起变更。
处理：禁止发起，提示当前状态不支持。

7. 交付超时
处理：工单保留待执行状态，并在列表和看板中标识超时。

### 八、审批流设计

#### 1. MVP 审批流

1. 申请人提交工单。
2. IT 资源管理员审批。
3. 运维执行人执行。
4. 工单完成。

#### 2. 审批原则

1. `APPLY`：重点审需求合理性、资源可用性、项目归属。
2. `CHANGE`：重点审变更必要性、风险、影响面。
3. `RECYCLE`：重点审是否已完成数据清理、是否允许解绑。

#### 3. Future 审批增强

1. 增加部门负责人会签。
2. 增加高成本资源二级审批。
3. 增加超时催办和 SLA。
4. 增加批量审批。

### 九、状态机

#### 1. 资源状态机

建议统一为以下状态：

1. `IDLE`
含义：已登记，可分配，未被项目占用。

2. `PENDING_DELIVERY`
含义：审批已通过，待交付。

3. `IN_USE`
含义：已交付，正在使用。

4. `CHANGING`
含义：变更执行中。

5. `PENDING_RECYCLE`
含义：回收审批已通过，待执行。

6. `RECYCLED`
含义：已回收，不可直接使用。

#### 2. 监控状态机

1. `NORMAL`
2. `WARNING`
3. `CRITICAL`
4. `UNKNOWN`

#### 3. 工单状态机

1. `DRAFT`
2. `PENDING`
3. `REJECTED`
4. `PENDING_EXECUTION`
5. `COMPLETED`

说明：`CANCELLED` 建议作为后续项，不强制进入 MVP。

### 十三、验收标准

#### 1. MVP 验收标准

1. 支持资源台账新增、编辑、查询、删除或停用。
2. 支持按资源类型、状态、项目、责任人、IP 查询资源。
3. 支持发起资源申请、变更、回收工单。
4. 支持审批通过、拒绝。
5. 支持执行交付、执行变更、执行回收。
6. 支持资源状态与工单状态联动。
7. 支持查看每个资源的责任归属、配置摘要、项目归属。
8. 支持查看工单全流程留痕。
9. 支持基础统计：资源总量、使用中数量、待审批数量、待执行数量、已回收数量。

#### 2. 发布阻断项

1. 资源编码重复校验缺失。
2. 工单审批或执行无留痕。
3. 状态流转可越级跳转。
4. 资源与工单关系断裂。
5. 关键筛选能力缺失导致无法管理台账。
