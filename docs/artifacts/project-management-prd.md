## 1. 背景与问题

信息化项目管理模块承担“项目台账、项目空间、过程资料、跨团队协同、验收申请、审核审批、结项归档”的统一管理职责。现有业务痛点主要集中在以下方面：

- 项目立项、执行、验收资料分散在线下目录、即时通讯和个人存储中，缺少统一项目空间
- 项目创建后无法自动按项目类型配置标准目录模板，导致资料结构不统一、归档难度高
- 项目与 IT资源、网络、数据、统一门户资源的关联关系不清，无法支撑全过程追踪
- 项目执行阶段缺少对资源状态、网络变更、数据使用与过程资料的集中记录
- 项目验收申请需要汇总 IT管理、数据管理、安全管理、统一门户管理、过程资料等多类材料，但当前缺少统一提交和审核入口
- 项目成员、观察员、评审专家等角色的查看、编辑、下载、删除权限边界不清，无法支撑文件夹/文档级精细控制
- 过程资料管理员缺少模板合规检查、归档管理的标准操作入口

因此，需要将“信息化项目管理”从简单项目登记升级为“全过程项目治理与资料归档中心”。

## 2. 产品目标

- 建立统一的项目台账，完整登记项目基本信息、项目类型、多团队负责人、目标用户、计划信息和资源需求
- 支持按项目类型自动配置标准文档目录模板到项目空间，并允许补充非标准目录模板
- 支持项目全过程过程资料上传、合规检查、归档与追踪，实现从立项到验收的资料闭环
- 建立项目与 IT资源、网络、数据、统一门户等资源对象的结构化关联，支撑执行与结项协同
- 提供项目验收申请与多环节审核审批能力，覆盖 IT管理、数据管理、安全管理、统一门户管理、过程资料等材料审核
- 提供项目统计分析能力，支撑项目数量、进度状态、资料完备度、验收进展、角色分布等管理视图
- 提供项目级、文件夹级、文档级细粒度角色权限控制

## 3. 范围

### 3.1 In Scope

- 模板管理
- 项目管理
- 项目验收申请
- 审核审批
- 统计分析
- 角色权限管理
- 过程资料合规检查与归档管理

### 3.2 Out of Scope

- IT资源、网络、数据、统一门户各模块自身主数据维护逻辑
- 安全测评、数据资产评估等外部专业活动的详细执行逻辑
- 非项目范围的企业级知识管理体系建设

## 4. 关键角色

- 项目经理：登记项目、维护计划与资源需求、组织执行、发起验收申请
- 项目成员：上传过程资料、维护执行记录、补充项目信息
- 观察员：只读查看项目状态、资料和关键进展
- 评审专家：参与验收评审，给出审查意见
- 过程资料管理员：维护模板、执行文档合规检查、管理归档
- IT管理审核人：审核 IT 管理相关验收材料与资源回收情况
- 数据管理审核人：审核数据使用、数据资产增量评估相关材料
- 安全管理审核人：审核网络回收、安全测评相关材料
- 统一门户管理审核人：审核统一门户相关材料
- 管理层：查看统计分析与项目治理总体情况

## 5. 现状与目标能力差距

### 5.1 项目台账维度

- 现状：项目信息登记颗粒度有限，负责人、目标用户、计划与资源需求没有形成统一结构
- 目标：形成“项目基础信息 + 多团队负责人 + 计划 + 目标用户 + 资源需求 + 阶段状态”的完整项目台账

### 5.2 项目空间维度

- 现状：资料目录依赖人工创建，项目之间结构不统一
- 目标：按项目类型自动下发标准目录模板，并支持非标准模板补充

### 5.3 过程资料维度

- 现状：过程资料上传、检查、归档没有统一入口
- 目标：全过程资料可上传、可校验、可归档、可追溯

### 5.4 验收治理维度

- 现状：验收材料收集和审核依赖人工汇总，缺少统一申请与审核闭环
- 目标：支持跨条线材料汇集、审核、整改、复核、结项

### 5.5 权限治理维度

- 现状：项目参与角色权限边界模糊，无法细化到文件夹和文档
- 目标：支持项目经理、成员、观察员、评审专家等角色的细粒度查看、编辑、下载、删除控制

## 6. 关键业务流程

### 6.1 模板管理流程

1. 过程资料管理员进入模板管理
2. 新建或维护标准目录模板，定义适用项目类型、目录层级、必传材料要求
3. 可补充非标准目录模板，用于特殊项目附加目录
4. 模板发布后，可被项目创建流程引用

### 6.2 项目创建与启动流程

1. 项目经理新建项目
2. 填写项目基础信息、项目类型、多团队负责人、项目计划、资源需求、目标用户
3. 系统依据项目类型自动配置标准目录模板到项目空间
4. 项目经理可按需追加非标准目录模板
5. 项目建立与 IT资源、网络、数据、统一门户资源的关联
6. 项目进入启动或执行状态

### 6.3 项目执行与资料归档流程

1. 项目成员和项目经理在项目空间内上传过程资料
2. 执行阶段持续记录资源状态、网络变更、数据使用情况和过程资料
3. 过程资料管理员进行文档合规检查，标记通过、整改、归档状态
4. 项目空间保留全过程资料版本和归档痕迹

### 6.4 项目验收申请流程

1. 项目经理发起项目验收申请
2. 系统汇总或引用 IT管理、数据管理、安全管理、统一门户管理、过程资料等材料项
3. 各审核角色按职责进行审核，给出通过、退回整改、补充材料等结论
4. 项目经理根据反馈补充资料后可再次提交
5. 全部审核通过后，项目进入结项归档阶段

### 6.5 项目结项流程

1. 项目经理确认结项
2. 系统记录资源监测与回收、网络回收与安全测评、数据资产增量评估、知识归档结果
3. 项目资料完成最终归档
4. 项目状态进入已结项或已归档

## 8. 状态模型

### 8.1 项目状态

- `INIT`：拟立项
- `STARTING`：启动中
- `EXECUTING`：执行中
- `ACCEPTANCE_REVIEW`：验收审核中
- `RECTIFICATION`：整改中
- `CLOSING`：结项处理中
- `ARCHIVED`：已归档

### 8.2 模板状态

- `DRAFT`：草稿
- `ACTIVE`：启用
- `DISABLED`：停用

### 8.3 资料合规状态

- `PENDING_CHECK`：待检查
- `PASSED`：合规通过
- `REJECTED`：不合规待整改
- `ARCHIVED`：已归档

### 8.4 验收申请状态

- `DRAFT`：草稿
- `SUBMITTED`：已提交
- `UNDER_REVIEW`：审核中
- `REJECTED`：退回整改
- `APPROVED`：审核通过
- `CLOSED`：验收完成

## 9. 业务规则

- 每个项目必须具备唯一项目编号
- 每个项目必须维护至少一名项目经理和一名业务或技术负责人
- 项目创建后必须依据项目类型自动生成标准目录模板
- 非标准目录模板只能在标准目录已生成的基础上追加，不能替代标准模板
- 必传目录或必传材料未满足前，不允许提交正式验收申请
- 项目验收申请必须覆盖 IT管理、数据管理、安全管理、统一门户管理、过程资料五类材料
- 过程资料管理员可对目录和文档执行合规检查与归档操作，但不替代业务审核结论
- 项目结项前必须补充资源回收、网络回收与安全测评、数据资产增量评估、知识归档结果
- 观察员默认只读，不得编辑、删除、归档资料
- 文件夹和文档权限应在项目级授权基础上支持单独收敛或扩展

## 10. 非功能与体验要求

- 项目列表支持按项目名称、项目类型、状态、负责人、验收状态、计划时间筛选
- 项目详情需按页签组织信息，避免立项、执行、验收、权限信息混杂
- 项目空间需让用户快速识别“缺资料、待检查、待整改、已归档”状态
- 验收申请页面需清晰区分不同审核条线的当前结论和待办状态
- 统计分析页面需支持按项目类型、状态、单位、负责人、时间区间切换
- 权限配置页面需能直观看到项目级、文件夹级、文档级权限差异

## 11. 验收标准

- 能创建项目并登记项目类型、多团队负责人、项目计划、目标用户、资源需求
- 能根据项目类型自动配置标准文档目录模板到项目空间
- 能为项目追加非标准目录模板并用于上传过程资料
- 能在执行阶段记录资源状态、网络变更、数据使用与过程资料
- 能发起项目验收申请并覆盖 IT管理、数据管理、安全管理、统一门户管理、过程资料五类材料
- 能由相关审核角色完成审核、退回整改、再次提交和最终通过
- 能在结项阶段记录资源监测回收、网络回收与安全测评、数据资产增量评估、知识归档
- 能基于项目经理、成员、观察员、评审专家等角色控制项目、文件夹、文档级权限
- 能提供项目统计分析视图，支持项目数量、阶段状态、验收进度、资料合规情况等查询

## 12. 待架构澄清项

- 项目空间是复用现有文件管理能力，还是为项目管理单独建模
- 标准模板、非标准模板与项目目录实例之间的映射方式如何设计
- 项目与 IT资源、网络、数据、统一门户资源的关联是统一关系模型还是分域引用
- 验收审核是否采用统一任务中心模型承载，还是按项目模块独立存储
- 文件夹/文档级权限是否与现有系统角色权限做叠加计算
